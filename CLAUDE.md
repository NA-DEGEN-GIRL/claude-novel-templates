## 3. 집필 워크플로우

모든 에피소드 작성은 다음 파이프라인을 따른다.

### 3.1 사전 준비 (Prep)

1. **이전 에피소드 확인**: 직전 에피소드 + 요약 파일을 읽는다.
2. **설정 파일 확인**: 변경된 설정이 있는지 `settings/` 폴더를 확인한다.
3. **연속성 체크**: 현재 시점의 캐릭터 상태, 위치, 미해결 떡밥을 정리한다.
4. **관계/정보 확인**: 등장 캐릭터들의 만남 이력(`relationship-log.md`), 정보 보유 현황(`knowledge-map.md`), 미이행 약속(`promise-tracker.md`)을 확인한다.
5. **에디터 피드백 확인**: `EDITOR_FEEDBACK.md`가 존재하면, `summaries/editor-feedback-log.md`와 대조하여 **미처리 피드백**이 있는지 확인한다. 미처리 항목 중 이번 화에 반영할 수 있는 합리적 제안이 있으면 집필 시 참고한다. 비합리적이거나 설정과 충돌하는 제안은 건너뛴다. 확인 결과를 `editor-feedback-log.md`에 기록한다.

### 3.2 집필 (Write)

1. 에피소드 구조(`settings/02-episode-structure.md`)에 따라 작성한다.
2. 문체 가이드(`settings/01-style-guide.md`)를 준수한다.
3. 목표 분량: {{TARGET_LENGTH}} (예: 4000~6000자)
4. **숫자·날짜·시간 계산 시 반드시 `novel-calc` MCP 도구를 사용한다.** 암산/추정 금지.
   - 날짜/요일: `date_calc`, `weekday`, `d_plus`, `date_diff`
   - 산술/수익률: `calculate`
   - 속도/거리/시간: `speed_distance_time`, `travel_estimate` (이동 수단 프리셋)
   - 전통 단위(리, 장, 척, 시진, 경, 근 등): `unit_convert`
   - 십이지시 ↔ 현대 시각: `convert_time`
   - 화폐(냥, 전, 푼, 금): `currency_calc`
   - 군량/보급 계산: `supply_calc`
   - 복리/성장 계산: `growth_calc`
   - 글자 수 확인: `char_count` (에피소드 분량 목표 체크)
5. **한자어 작명 시 반드시 `novel-hanja` MCP 도구를 사용한다.** LLM 추론으로 한자를 조합하지 말 것.
   - 한자어 풀이: `hanja_lookup` (한자 → 음/훈/획수 분석)
   - 한글 음으로 한자 검색: `hanja_search` (음/뜻 힌트로 한자 찾기)
   - 의미로 한자 검색: `hanja_meaning` (키워드로 관련 한자 탐색, 예: "fire", "바람")
   - 한자어 검증: `hanja_verify` (글자 존재 여부 + 소설 내 중복 체크)
6. **한자 표기는 용어 첫 등장 시에만 병기한다.**
   - 처음 등장하는 용어: `내공(內功)` 형태로 한글(漢字) 병기
   - 이미 등장한 용어: `내공` 한글만 표기
   - `summaries/hanja-glossary.md`를 참조하여 이미 소개된 용어인지 확인한다
   - 새 용어를 소개한 경우, 후처리 단계에서 용어집에 추가한다
   - **예외 재표기**: 30화 이상 미등장 후 재등장, 또는 동음이의어 구분 필요 시
7. **작품의 시대·세계관에 맞는 단위와 용어를 사용한다.**
   - 나레이션과 대사에서 세계관에 존재하지 않는 도량형·외래어를 쓰지 않는다
   - **비현대 배경**(무협, 판타지, 사극 등): cm/km/kg 등 미터법 → 전통 단위(리, 장, 척, 근 등) 또는 세계관 고유 단위 사용. `unit_convert` 도구로 변환. 영어 외래어(시스템, 패턴, 에너지 등) → 한자어·고유어로 대체
   - **현대/SF 배경**: 현대 단위·용어 사용 가능
   - 판단 기준은 `settings/04-worldbuilding.md`의 시대·배경 설정을 따른다
   - 소설별로 더 상세한 규칙이 있으면 해당 소설의 CLAUDE.md가 우선한다 (예: 001의 외래어 교체 목록)
   - **비현대 배경의 숫자 표기**:
     - 본문(나레이션·대사)에서 **아라비아 숫자를 사용하지 않는다**. 한글 수사(고유어/한자어)로 쓴다.
     - **소수점 표현 금지**: `1.5장` → `한 장 반`, `1.7장` → `한 장 칠 척`. 아라비아 소수점(`.`)은 비현대 세계관에 존재하지 않는 표기법이다.
     - 수량: `100명` → `백여 명`, `5명` → `다섯 명`, `30` → `서른`
     - 서수/순번: `3번째` → `셋째` 또는 `세 번째`
     - 기간: `3일` → `사흘`, `7일` → `이레`, `10일` → `열흘`, `15일` → `보름`
     - 큰 수: `1만` → `만`, `10만` → `십만`
     - **예외**: EPISODE_META, plot/, summaries/ 등 메타 영역에서의 아라비아 숫자 사용은 허용
8. **각주가 필요한 소설은 `[^N]` 포맷을 사용한다.** 본문 내 참조: `[^1]`, 각주 정의: `[^1]: 텍스트`. 리더에서 클릭 가능한 링크로 자동 변환된다. `\[N\]` 또는 `[N]` 형식을 사용하지 않는다.

### 3.3 자체 검토 (Review)

작성 완료 후, `.claude/agents/reviewer.md`의 기준으로 자체 검토한다.

| 항목 | 기준 | 최소 점수 |
|------|------|-----------|
| 문체 일관성 | 설정된 문체 가이드 준수 | 4/5 |
| 캐릭터 일관성 | 말투, 성격, 행동 패턴 | 4/5 |
| 구조 완성도 | 에피소드 구조 준수 | 4/5 |
| 엔딩 훅 | 다음 화 기대감 | 3/5 |
| 감정선 | 적절한 감정 변화 곡선 | 3/5 |
| 몰입도/페이싱 | 지루한 구간 없음 | 4/5 |
| 복선/떡밥 | 회수 또는 새 떡밥 설치 | 3/5 |

→ 하나라도 기준 미달이면 수정 후 재검토.

자체 검토와 동시에 `.claude/agents/gemini-feedback.md` 에이전트를 호출하여 **Gemini 편집 리뷰를 병렬로 수행**한다.

→ 자체 검토 + Gemini 리뷰 결과를 종합하여 텍스트 수정을 먼저 반영한 후,
  `.claude/agents/korean-proofreader.md` 기준으로 한글 교정을 수행한다 (맞춤법, 번역투, AI 습관 단어, 줄임표 등).

> **순서 근거**: Gemini가 텍스트 수정을 제안하면 교정 대상이 바뀌므로, 한글 교정은 반드시 모든 텍스트 수정이 완료된 후 마지막에 수행한다.

### 3.4 연속성 검증 (Continuity Check)

`.claude/agents/continuity-checker.md`의 기준으로 검증한다.

| 항목 | 체크 내용 |
|------|-----------|
| 위치 연속성 | 캐릭터가 물리적으로 이동 가능한 위치에 있는가 |
| 부상/상태 | 이전 부상, 피로, 감정 상태가 반영되었는가 |
| 능력/레벨 | 파워 레벨이 설정과 일치하는가 |
| 시간선 | 시간 흐름이 자연스러운가 (`novel-calc`의 `date_diff`, `d_plus`로 검증) |
| 대화 톤 | 캐릭터별 말투가 일관적인가 |
| 고유명사 | 이름, 지명, 기술명 등이 정확한가 |
| 사망 캐릭터 | 죽은 캐릭터가 재등장하지 않는가 |
| 미해결 떡밥 | 이전에 설치한 떡밥과 충돌하지 않는가 |
| 약속/계획 | 미이행 약속이 무시되지 않았는가, 시한이 맞는가 (`promise-tracker.md`) |
| 정보 보유 | 캐릭터가 아직 모르는 정보를 아는 것처럼 말하지 않는가 (`knowledge-map.md`) |
| 만남/관계 | 만난 적 없는 캐릭터를 아는 것처럼 대하지 않는가 (`relationship-log.md`) |

→ 위반 사항 있으면 수정.

### 3.5 후처리 (Post-Process)

1. **요약 작성**: `summaries/` 폴더에 에피소드 요약을 작성한다.
   - 요약 작성 후 `.claude/agents/summary-validator.md`의 단건 검증으로 원문 대조 가능. 특히 relationship-log, knowledge-map의 방향성·귀속 정확성을 확인.
2. **연속성 추적 갱신**: 아래 3개 파일을 이번 화 내용에 맞게 갱신한다.
   - `summaries/promise-tracker.md` — 새 약속 추가, 기존 약속 진전/완료 처리, 시한 확인
   - `summaries/knowledge-map.md` — 캐릭터가 새 정보를 습득했으면 갱신, 정보 전파가 있었으면 기록
   - `summaries/relationship-log.md` — 새로운 만남/대화/관계 변화가 있었으면 갱신
3. **config.json 업데이트**: `/root/novel/config.json`에 새 에피소드를 등록한다.
   - 해당 part의 `episodes` 배열에 `{ "number": N, "title": "제목", "file": "경로" }` 추가
   - **episodes 배열은 반드시 number 기준 오름차순 정렬** 유지
   - `totalEpisodes` 값은 **실제 등록된 에피소드 수**와 일치시킨다 (최신 화 번호가 아님)
   - 에피소드가 비순차적으로 생성되더라도, 삽입 후 정렬하여 순서를 보장한다
   - **누락 방지**: 등록 전 직전 등록된 에피소드 번호를 확인하고, 사이에 파일은 존재하지만 미등록된 에피소드가 있으면 반드시 함께 등록한다. 번호가 점프하면 독자에게 에피소드가 보이지 않는다.
   - **date 연동**: EPISODE_META의 `date` 필드가 리더에서 해당 에피소드의 업로드 날짜로 사용된다. 에피소드 등록 시 EPISODE_META에 date가 기입되어 있는지 함께 확인한다.
4. **EPISODE_META 삽입** (선택): 에피소드 끝에 메타데이터 블록을 추가한다.
   - EPISODE_META의 `date` 필드에 **에피소드 집필 완료일**(오늘 날짜)을 `"YYYY-MM-DD"` 형식으로 기입한다.
   - 이 date 값은 리더에서 에피소드의 **업로드 날짜**로 표시된다.
5. **Gemini 피드백 로그 갱신**: 3.3단계에서 처리한 Gemini 피드백의 반영/참고/건너뜀 결과를 `summaries/editor-feedback-log.md`에 기록한다. (Gemini 리뷰 자체는 3.3에서 이미 수행됨)
6. **Git 커밋**: 집필 진행 상황을 기록하기 위해 적절한 시점에 커밋한다.
   - 에피소드 집필 완료 후 (본문 + 요약 파일 함께)
   - config.json 업데이트 후 (에피소드 등록, 소설 설정 변경)
   - 여러 화를 연속 집필할 때는 2~3화 단위로 묶어서 커밋 가능
   - 커밋 메시지: `{소설명} {N}~{M}화 집필` 또는 `config.json: {변경 요약}`
   - **push는 하지 않는다** — push/배포는 사용자가 명시적으로 요청할 때만 수행

### 3.6 병렬 집필 후 정합성 점검 (Parallel Reconciliation)

에피소드를 병렬로 집필한 경우(여러 에이전트가 동시에 다른 화를 작성), **집필 완료 후** 아래 절차를 반드시 수행한다.

#### 병렬 집필 판단 기준

아래 중 하나라도 해당하면 "병렬 집필"로 간주한다:
- 여러 에이전트가 동시에 다른 화를 집필했음
- 요약 파일(running-context 등)이 집필 전후로 갱신되지 않은 화가 연속 존재함
- 사용자가 명시적으로 병렬 집필임을 알려줌

#### 병렬 집필 시 각 에이전트의 규칙

- **3.1~3.3은 그대로 수행** (맥락 로딩, 집필, 자체 검토)
- **3.4 연속성 검증**: 이전 에피소드(자기 담당 범위 내)에 대해서만 수행
- **3.5 후처리에서 요약 갱신은 보류** — 다른 에이전트와 충돌하므로 건너뛴다
- config.json 등록, EPISODE_META 삽입은 수행한다

#### 병렬 집필 완료 후 정합 절차

1. **교차 연속성 검증**: 병렬로 작성된 에피소드들을 **번호순으로 통독**하며 에피소드 간 연속성을 점검한다. `continuity-checker`의 "병렬 교차 검증" 모드를 사용한다.
   - 에피소드 간 장소/시간 연결이 자연스러운가
   - 동일 캐릭터의 상태(감정, 부상, 위치)가 화 사이에서 모순 없는가
   - 같은 사건을 다르게 서술하지 않았는가
   - 엔딩 훅과 다음 화 시작이 매끄럽게 이어지는가
2. **충돌 수정**: 발견된 불일치를 수정한다 (번호가 낮은 화를 기준으로 높은 화를 조정).
3. **요약 일괄 갱신**: 번호순으로 요약 파일을 일괄 갱신한다.
   - running-context, character-tracker, episode-log
   - promise-tracker, knowledge-map, relationship-log

### 3.7 정기 점검 (Periodic Checkpoint)

연속 집필 시 **5화마다** 아래 점검을 수행한다. (5, 10, 15, 20화… 완료 시점)

> 이 점검은 에피소드 단위 파이프라인(3.1~3.5)으로 잡히지 않는 **누적 드리프트**를 방지한다.

#### 트리거 조건

- 직전 정기 점검 이후 5화 이상 집필했을 때
- 아크가 전환되는 시점 (아크 종료 시에는 5화 미만이어도 수행)

#### 점검 항목

| # | 항목 | 방법 |
|---|------|------|
| P1 | 요약 정합성 | `summary-validator` 일괄 감사(모드 B) — 직전 점검 이후 작성된 에피소드 대상 |
| P2 | 복선 회수 시한 | `plot/foreshadowing.md`에서 회수 예정 시점이 지난 복선이 있는지 확인 |
| P3 | 캐릭터 상태 최신성 | `character-tracker`의 현재 상태가 최신 에피소드 종료 시점과 일치하는지 확인 |
| P4 | 캐릭터 성격 드리프트 | 최근 5화의 주요 캐릭터 대사/행동이 `settings/03-characters.md`의 성격 설정과 부합하는지 통독 검토 |
| P5 | 미이행 약속 점검 | `promise-tracker`에서 시한이 지났거나 방치된 약속이 있는지 확인 |
| P6 | running-context 갱신 | `running-context.md`가 200줄 이내인지, 최신 상태를 정확히 반영하는지 확인 |
| P7 | Gemini 일괄 리뷰 | 직전 점검 이후 에피소드를 `gemini-feedback` 배치 모드(모드 B)로 일괄 리뷰. 텍스트 수정 제안을 반영한다 |
| P8 | 한글 품질 일괄 점검 | P7 반영 후 `korean-proofreader` 기준으로 재검수. 불규칙 활용, 수사 혼용, 번역투, AI 습관 단어 등 누적 오류를 일괄 교정 |
| P9 | 메타 참조 금지 점검 | 본문(나레이션·대사·독백)에서 "X화에서", "지난 화", "앞으로 N화" 등 에피소드 번호로 과거/미래를 지칭하는 표현이 없는지 전수 검사. 발견 시 날짜·장소·사건명으로 대체 |

#### 점검 후 조치

1. 발견된 이슈를 수정한다 (요약 오류 → 즉시 수정, 복선 누락 → 향후 전개에 반영 계획 기록).
2. `running-context.md`를 최신 상태로 갱신한다.
3. 점검 결과를 커밋에 포함한다: `{소설명} {N}~{M}화 정기 점검 완료`

## 5. 금지 사항

> 아래 항목은 어떤 상황에서도 위반하지 않는다.

1. **캐릭터 성격 급변 금지**: 점진적 변화만 허용. 갑작스러운 성격 변화 시 반드시 납득 가능한 사건이 선행되어야 한다.
2. **데우스 엑스 마키나 금지**: 위기 해결은 이전에 설정된 능력/자원/인물로만 가능하다.
3. **설정 모순 금지**: 한번 확립된 세계관 규칙은 변경 불가. 변경이 필요하면 설정 파일을 먼저 수정한다.
4. **분량 과부족 금지**: 목표 분량의 ±20% 이내를 유지한다.
5. **메타적 표현 금지**: "소설에서처럼", "독자 여러분" 등 4th wall breaking 금지.
6. **고유명사 변경 금지**: 설정 파일에 정의된 이름을 임의로 변경하지 않는다.
7. **"X화" 메타 참조 금지**: 본문(나레이션·대사·독백) 안에서 "3화에서", "56화에서 알게 된" 등 에피소드 번호로 과거를 지칭하지 않는다. 캐릭터는 자기 이야기가 '화'로 나뉘어 있다는 사실을 모른다. 과거 사건은 **날짜, 장소, 사건명** 등 작중 정보로 지칭한다. ※ EPISODE_META, plot/, summaries/ 등 메타 영역에서의 화 번호 사용은 허용.
8. **시점 전환 스포일러 금지**: `[시점 전환: 세르반]` 같은 메타 태그를 본문에 노출하지 않는다. 독자가 아직 모르는 캐릭터 이름이 드러나면 스포일러이고, 이미 아는 이름이라도 몰입을 깨뜨린다. 시점 전환은 `***` 장면 구분 후 **서술 자체(톤·장소·감각)로 자연스럽게** 드러낸다.

## 7. 표지 이미지

소설 루트에 `cover.png`(또는 `.jpg`)를 저장하고, `/root/novel/config.json`의 `cover` 필드에 경로를 등록한다.

**자동 생성**: NovelAI MCP 서버의 `generate_image` tool로 생성한다.
- 첫 에피소드 집필 시 `cover.png`가 없으면 자동 생성
- 이미 존재하면 건드리지 않는다 (사용자가 명시적으로 요청할 때만 재생성)

**프롬프트 작성 요소:**
- 주인공의 외형 (`settings/03-characters.md`의 외모 설정)
- 소설의 장르와 분위기 (무협→동양풍, 현대→도시/네온, SF→미래적)
- 핵심 상징물 (검, 마법, 기술 장치 등)
- 톤앤무드 (다크→어두운 조명, 밝음→선명한 색감)
- 표지에 적합한 구도: 상반신 또는 전신, 임팩트 있는 포즈
- 권장 해상도: 832x1216 (세로형)

생성 후:
1. `config.json`의 해당 소설 `cover` 필드를 `"{{NOVEL_ID}}/cover.png"`로 업데이트한다.
2. **사용한 프롬프트를 `cover-prompt.txt`에 저장한다** (재생성/수정 시 기준점).

**`cover-prompt.txt` 형식:**

```
[Prompt]
masterpiece, best quality, amazing quality,
1boy, solo, upper body, ...태그...

[Negative Prompt]
lowres, bad anatomy, ...

[Settings]
resolution: 832x1216
tool: generate_cover / generate_image
date: YYYY-MM-DD
```

- 표지를 재생성할 때는 이 파일의 프롬프트를 기반으로 수정한다.
- 프롬프트 없이 표지가 존재하면, 역으로 프롬프트를 추론하여 `cover-prompt.txt`를 작성한다.

### character-prompts.md

소설 루트에 `character-prompts.md` 파일을 생성하여 캐릭터별 이미지 생성 프롬프트를 관리한다.
NovelAI MCP 서버(`novelai-image`)가 이 파일을 파싱하여 자동으로 이미지를 생성한다.

**캐릭터 초상화**: `generate_character` tool로 생성한 이미지는 `assets/characters/`에 저장된다. 캐릭터의 기준 외형을 확인하는 참고 자료로 사용한다.

**초상화 관리 규칙:**
- 캐릭터 첫 등장 전에 `generate_character`로 기본 초상화를 생성해 둔다
- 외모가 크게 변하는 경우 (변신, 시간 경과, 변장 등) 해당 변형의 초상화도 생성한다
- 변형 초상화 파일명: `{캐릭터명}-{변형명}.png` (예: `사쿠라-방송_모드.png`)

## 8. 에피소드 삽화

### 삽입 기준

매 에피소드가 아닌, **아래 상황에서만** 삽화를 삽입한다:

- 캐릭터 첫 등장 또는 외모 변화 (변장, 부상, 새 의상 등)
- 감정적으로 인상적인 장면 (결투 직후, 고백, 이별 등)
- 새로운 장소/세계관 요소가 처음 등장할 때
- 에피소드 전체 톤을 시각적으로 강조하고 싶을 때

> 매 화 넣지 않는다. 정말 시각적으로 강한 장면일 때만 넣어야 효과가 있다. 빈도는 자유롭게 판단하되, 남발하지 않는다.

### 삽화 생성 방법

**반드시 `generate_illustration` tool을 사용한다** (캐릭터 일관성 자동 보장):

```
generate_illustration(
  novel_id="{{NOVEL_ID}}",
  scene_prompt="장면/배경/분위기 태그 (캐릭터 외모 태그는 넣지 않는다)",
  characters=[
    {"name": "캐릭터A"},
    {"name": "캐릭터B", "variant": "변형명", "extra_tags": "장면 전용 추가 태그"}
  ],
  output_filename="chXXX-scene-name.png"
)
```

**동작 원리:**
1. `character-prompts.md`에서 캐릭터 태그를 자동 로드하여 V4 char_captions로 전달
2. 캐릭터별 좌표 지정으로 다인물 구도에서 정확한 배치
3. 장면/배경 태그와 캐릭터 외모 태그가 분리되어 속성 혼동 방지

> **⚠️ `generate_image`는 에피소드 삽화에 사용하지 않는다.**
> `generate_image`는 캐릭터 태그를 자동 로드하지 않으므로 캐릭터 외모가 일관되지 않는다.

### 삽화 형식 (필수 — 이 형식만 허용)

에피소드 본문 끝(EPISODE_META 직전)에 **반드시 아래 blockquote 형식으로** 삽입한다:

```markdown
---

> **삽화**: {{캐릭터명}} — {{상황 설명}}
> ![삽화]({{이미지 경로}})
```

> **⚠️ 필수**: 반드시 `>` blockquote + `**삽화**:` 설명 라인을 포함해야 한다.
> 리더가 이 형식을 파싱하여 삽화 설명을 표시한다.
> `![설명](path)` 만 단독으로 쓰는 것은 **금지** — 설명이 표시되지 않는다.

- 이미지 경로는 `{{NOVEL_ID}}/assets/illustrations/` 하위.
- `character-prompts.md`에 해당 캐릭터의 프롬프트가 없으면 먼저 추가한다.
- `config.json`에서 해당 에피소드 항목에 `"ill": true` 추가 (리더에서 삽화 배지 표시용)

### 삽화 추적 (필수)

삽화를 생성할 때마다 `summaries/illustration-log.md`에 기록한다:
- 에피소드 번호, 파일명, 등장 캐릭터, scene_prompt, 생성일
- 이 기록이 없으면 재생성/감사 시 일관성 유지가 불가능하다

### 삽화 관리

`.claude/agents/illustration-manager.md`로 삽화를 체계적으로 관리한다:
- **단건 검증**: 삽화 삽입 직후 이미지-본문 일치, 태그 정합성, 형식 등 6항목 검증
- **일괄 감사**: 아크 종료 시 전체 삽화 점검 (외모 변화 반영, 고아 이미지, 깨진 참조 등 11항목)
- **재생성**: 감사에서 문제 발견 시, 기존 프롬프트 기록 기반으로 수정 후 재생성

## 9. 대화 관계 규칙

### 8.1 호칭/어투 매트릭스

캐릭터 간 대화 시 존댓말/반말/호칭을 정의한다. "행(Row) 캐릭터가 열(Column) 캐릭터에게 말할 때"의 규칙이다. 자주 대화하는 쌍만 기록한다.

| 화자 → 청자 | {{CHAR_A}} | {{CHAR_B}} | {{CHAR_C}} |
|-------------|-----------|-----------|-----------|
| **{{CHAR_A}}** | — | {{존/반+호칭}} | {{존/반+호칭}} |
| **{{CHAR_B}}** | {{존/반+호칭}} | — | {{존/반+호칭}} |
| **{{CHAR_C}}** | {{존/반+호칭}} | {{존/반+호칭}} | — |

> 미등록 조합은 기본값 적용: {{DEFAULT_RULE}} (예: "처음 만나는 사이 = 존댓말")

### 8.2 상황별 어투 전환

동일 캐릭터 쌍이라도 상황에 따라 어투가 달라지는 경우를 정의한다.

| 캐릭터 | 상황 A | 상황 B |
|--------|--------|--------|
| {{CHAR}} | {{공적 석상: ~합니다}} | {{사적 대화: 반말}} |

### 8.3 어투 변화 이력

스토리 진행에 따라 어투가 바뀐 경우를 기록한다. 이 이력이 현재 상태의 근거가 된다.

| 화수 | 화자 → 청자 | 변화 내용 | 사유 |
|------|------------|----------|------|
| {{N}}화 | {{A}} → {{B}} | 존댓말 → 반말 | {{사건 설명}} |

### 8.4 어투 원칙

1. **어투 변화는 사건에 의해서만 발생한다**: 설명 없이 갑자기 존댓말↔반말이 바뀌지 않는다.
2. **감정 극단 시 일시적 이탈은 허용한다**: 극도의 분노/공포에서 평소와 다른 어투를 쓸 수 있지만, 장면 종료 후 원래로 돌아간다.
3. **공적/사적을 구분한다**: 같은 캐릭터 쌍이라도 공적 석상과 사적 대화에서 어투가 다를 수 있다.
4. **어투 변화는 원칙적으로 역행하지 않는다**: 반말로 전환된 관계가 다시 존댓말로 돌아가려면 특수 사건(배신, 관계 단절 등)이 필요하다.

## 10. 에디터 피드백 시스템

외부 AI 에디터가 `EDITOR_FEEDBACK.md`에 피드백을 작성한다. 이 파일은 수시로 갱신되며, 집필 에이전트가 참고한다.

### 10.1 피드백 처리 원칙

1. **합리적 제안은 반영한다**: 문체 개선, 연속성 오류 지적, 캐릭터 일관성 문제 등.
2. **설정과 충돌하는 제안은 건너뛴다**: 금지 사항, 세계관 규칙, CLAUDE.md 원칙에 어긋나는 제안은 사유를 기록하고 건너뛴다.
3. **모든 처리 결과를 로그에 기록한다**: 반영/건너뜀/참고 여부와 사유를 남긴다.
4. **재검토 의견은 재고한다**: 에디터가 `[재검토 요망]` 의견을 남긴 경우, 저자는 다시 한번 대안을 고민한다. 동의하면 반영하고, 여전히 이견이 있으면 사유를 기록하고 건너뛴다.

### 10.2 처리 로그 (`summaries/editor-feedback-log.md`)

피드백을 확인할 때마다 처리 결과를 기록한다. 이 로그로 "어디까지 봤는지"를 추적한다.

> **타임스탬프**: `EDITOR_FEEDBACK.md`에 시간(`HH:MM`)이 포함된 경우, 로그의 날짜·피드백 섹션에도 동일하게 시간을 포함하여 1:1 대응이 가능하게 한다.

```markdown
# 에디터 피드백 처리 로그

| 날짜 | 피드백 섹션 | 결과 | 비고 |
|------|------------|------|------|
| 2026-02-27 23:40 | [2026-02-27 23:40] 1화 §1 Language 개선제안 | ✅ 반영 | 6화 집필 시 반영 |
| 2026-02-27 23:40 | [2026-02-27 23:40] 1화 §2 Continuity 체크포인트 | 📌 참고 | 의도된 복선, 향후 전개에서 해명 예정 |
| 2026-02-27 23:40 | [2026-02-27 23:40] 1화 §3 Character 박준호 | ⏭️ 건너뜀 | 프롤로그 한정 인물, 설정 의도대로 |
```

**결과 표기:**
- `✅ 반영` — 제안을 수용하여 집필/수정에 적용
- `📌 참고` — 유효한 지적이나 즉시 반영이 아닌 향후 참고 (복선, 장기 계획 등)
- `⏭️ 건너뜀` — 설정 충돌, 비합리적 제안 등으로 미반영 (사유 필수)

### 10.3 워크플로우 연동

- **3.1 Prep**: `EDITOR_FEEDBACK.md` 존재 시 → `editor-feedback-log.md`와 대조 → 미처리 피드백 확인 → 이번 화 관련 항목 참고
- **3.5 Post-Process**: 확인한 피드백의 처리 결과를 `editor-feedback-log.md`에 기록
